# wikipedia-tool

This repository is an Anthropic prompt-engineering take-home project. It builds a Claude Haiku 4.5 agent that answers questions with a single `search_wikipedia(query)` tool, then evaluates whether the agent used the tool appropriately and answered correctly.

The project is meant to be easy to review: you can run a small demo in under a minute, inspect full saved runs without spending API money, and compare prompt iterations with statistical tests.

Useful companion docs:

- [`ITERATIONS.md`](./ITERATIONS.md) tells the full prompt-engineering story: each prompt revision, what it was trying to fix, and whether the metrics moved.
- [`PLANS.md`](./PLANS.md) is the original design doc and implementation scratchpad. The "CLAUDE AMENDMENTS" section contains the locked architecture, schemas, and contracts.

## What It Does

The agent is Claude Haiku 4.5 (`claude-haiku-4-5`) with one tool: `search_wikipedia(query)`. The tool talks directly to the MediaWiki Action API using a single-call `generator=search` + `prop=extracts` request, then caches results in SQLite at `cache/wikipedia.sqlite`.

The eval set has 452 questions:

- 384 **tool-required** questions about post-March-2025 TV and sports facts. These were generated by Opus 4.7 using the same Wikipedia tool and have golden answers.
- 68 **tool-not-required** questions about stable evergreen facts that Haiku should usually answer from pretraining.

Each agent trace is graded two ways:

- A programmatic grader checks whether the tool was called. Returns a boolean value.
- A date-aware Sonnet 4.6 judge checks whether the final answer matches the golden answer. The judge sees the answer, not the tool trace, so answer quality and tool call behavior are separate.

Run summaries include tool-use metrics, answer correctness on the tool-required slice, breakdowns by whether the tool was called and by category (`tv`, `sports`), McNemar's exact test for binary run comparisons, and a paired t-test for `n_tool_calls`.

## Setup

Requires Python 3.11+ and an Anthropic API key.

```bash
pip install -r requirements.txt
echo 'ANTHROPIC_API_KEY=sk-ant-...' > .env
```

The Wikipedia cache is created on first use. It is safe to delete `cache/wikipedia.sqlite` if you want a fresh cache.

To ask the Haiku agent any question you want:

```bash
python -m agent.haiku_agent "<your_prompt>"
```

## Quick Demo

To ask the Haiku agent a few sample questions:

```bash
python demo.py query
```

This chooses three stratified random examples from the eval set: one stable question, one TV question that should require search, and one sports question that should require search. It prints a readable transcript for each question, including the user prompt, tool calls, truncated tool results, and final answer. Output is colorized when run in a terminal.

For a slightly larger test:

```bash
python demo.py eval --limit 20
```

That runs a stratified random subset of 20 through inference and both graders, then prints a compact breakdown table. It exercises the agent, Wikipedia client, programmatic grader, LLM judge, and stats code without running the full 452-question eval.

Both demo commands accept prompt-version flags:

```bash
python demo.py query --seed 42 --tool-description-id v5 --system-prompt-id v2
python demo.py eval --limit 20 --seed 42 --tool-description-id v5 --system-prompt-id v2
```

If API access is inconvenient, you can inspect the saved outputs in `runs/`. Each `runs/<timestamp>__td-<tool_description_id>__sp-<system_prompt_id>/results.jsonl` file has one record per question with the full message history, tool-call trace, judge verdict, and judge reasoning. `summary.json` contains the aggregated metrics.

## Full Eval Pipeline

The eval datasets are already checked in. Regenerate it only if you want fresh questions:

```bash
python -m datagen.gen_tool_required \
    --seeds data/seeds.jsonl \
    --out data/tool_required.jsonl \
    --target 384

python -m datagen.gen_stable \
    --out data/tool_not_required.jsonl \
    --target 68
```

`gen_tool_required` runs 24 async Opus 4.7 calls in parallel. Each worker handles several seeds in a growing-context conversation. It writes partial progress to `<out>.partial.jsonl`, so interrupted runs keep completed examples.

To run inference and grading over all 452 questions:

```bash
python run.py \
    --tool-description-id v5 \
    --system-prompt-id v2 \
    --concurrency 24
```

At concurrency 24, a full live run takes roughly 3-4 minutes and costs a few dollars in API usage. Each run writes a directory named `runs/<timestamp>__td-<tool_description_id>__sp-<system_prompt_id>/`, which makes prompt versions self-documenting.

The run directory contains:

- `inference.jsonl`: one trace per question with the full message history, tool calls, final answer, latency, and `n_tool_calls`.
- `results.jsonl`: inference traces plus grader outputs such as `tool_called`, `tool_use_correct`, `answer_correct`, and `judge_reasoning`.
- `summary.json`: run-level metrics including the tool-use confusion matrix, answer-correctness breakdowns, and stable-set tool-call rate.

## Comparing Runs

Use `eval.compare` to check whether a prompt change likely improved behavior or just moved noise around:

```bash
# McNemar's exact test for a binary metric
python -m eval.compare runs/<run_a> runs/<run_b> --metric answer_correct

# Paired t-test for tool-call count
python -m eval.compare runs/<run_a> runs/<run_b> --metric n_tool_calls
```

Both commands accept either a run directory or a direct `results.jsonl` path. McNemar output includes `b`, `c`, exact `p`, and raw accuracy delta. The t-test output includes paired means, delta, `t`, `df`, and `p`.

For `n_tool_calls`, add `--conditional-on-called` to restrict the comparison to questions where both runs searched. This is useful for asking whether one prompt searched more efficiently after both prompts had already decided to use the tool.

## Prompt Iteration

Tool description and system prompt iterations are stored in `prompts/`:

```text
prompts/
  tool_descriptions/
    baseline.md
    v1.md
    v2.md
    v3.md
    v4.md
    v5.md
  system_prompts/
    baseline.md
    v1.md
    v2.md
```

The prompt ID is the filename without `.md`, so adding `prompts/tool_descriptions/v6.md` makes `--tool-description-id v6` available to `demo.py` and `run.py`.

System prompts support one placeholder: `{{TIMESTAMP}}`, which is replaced with today's UTC date at every agent invocation. That keeps relative-time questions such as "this year" grounded during both inference and judging.

Read [`ITERATIONS.md`](./ITERATIONS.md) for the full revision history, prompt diffs, statistical comparisons, and remaining failure modes.

## Project Map

```text
agent/        Haiku inference loop, prompt loader, tool-call tracking
wiki/         MediaWiki client and SQLite cache, shared by agent and datagen
datagen/      tool-required and stable eval-set generators
eval/         graders, summary stats, and run-comparison CLI
prompts/      versioned tool descriptions and system prompts
data/         seeds and generated eval datasets
cache/        Wikipedia SQLite cache, safe to delete
runs/         saved inference, grading, and summary outputs
tests/        pytest suites with mocked Anthropic and Wikipedia calls
demo.py       reviewer-friendly CLI for sample queries and small evals
run.py        full inference + grading pipeline
```

Start with these files if you want to inspect implementation details:

- `agent/haiku_agent.py`: the inference loop, prompt rendering, max-10 tool-call cap, and trace capture.
- `wiki/client.py`: the only place live Wikipedia requests happen.
- `eval/grade.py`: the programmatic grader and Sonnet judge prompt.
- `eval/stats.py`: summary aggregation for `summary.json`.
- `eval/compare.py`: McNemar exact test, paired t-test, and comparison CLI.
- `datagen/gen_tool_required.py`: async Opus generation for tool-required examples, including near-duplicate filtering.
- `datagen/gen_stable.py`: stable evergreen Q&A generation.

## Tests And Sanity Checks

The test suite is fully offline: Anthropic and Wikipedia are mocked.

```bash
pytest -q
```

Current coverage is 145 tests. Before spending money on a full eval, run:

```bash
pytest -q
python demo.py query --seed 42
python demo.py eval --limit 20 --seed 42
```

Those commands exercise the client, agent, data schemas, graders, stats, and comparison code at much lower cost than a full run.

## Design Rationale

The main design goal is to isolate prompt behavior from infrastructure surprises.

- The tool description and system prompt are independently versioned, with `baseline` anchors for clean ablations.
- The tool returns one lead-section result per call (`k=1`), plus title, URL, lead text, and error fields. Multi-hop behavior is allowed through repeated calls, capped at 10.
- The eval topic scope is intentionally narrow: post-March-2025 TV and sports. That reduces pretraining contamination while avoiding topics likely to trigger policy refusals.
- Answer correctness is reported for the tool-required slice. Stable questions are mainly used to measure over-searching, which appears in the tool-use confusion matrix and `tool_not_required` summary block.
- The LLM judge sees only the final answer, not the tool trace. Tool-use behavior is measured programmatically, while answer correctness is judged against the golden answer.
